## Project

Multi-Mode Card Framework — a Unity 6.3 LTS card game engine (Blackjack / Solitaire / Texas Hold'em) targeting WebGL, PC, Mobile, and Meta Quest 3. Active solo portfolio project. The authoritative status and roadmap live in `PROJECT_ROADMAP.md` (current epic/task state) and `BACKLOG.md` (story-pointed backlog). Read those before proposing scope changes.

## Architecture (Big Picture)

Strict layered MVC + VContainer DI. The cardinal rule: **Core is pure C# POCO, no `UnityEngine` references, no MonoBehaviours.** Everything else depends on Core; Core depends on nothing.

```
┌─────────────────────────────────────────┐
│ Presentation (Views, Controllers)       │ MonoBehaviour + IStartable/IDisposable
└────────────────────┬────────────────────┘
                     │ DI (VContainer)
┌────────────────────▼────────────────────┐
│ Cloud (PlayFab)  /  Input  /  XR        │ Platform adapters, async APIs
└────────────────────┬────────────────────┘
                     │ references
┌────────────────────▼────────────────────┐
│ Core (CardData, Deck, *Engine, Utils)   │ Pure C# — testable in EditMode
└─────────────────────────────────────────┘
```

### Assembly definitions (`Assets/Project/Scripts/*/  *.asmdef`)

| AsmDef       | Role                                  | Notes |
|--------------|---------------------------------------|-------|
| `Core`       | Pure C# game logic                    | No references; keep it that way |
| `Presentation` | Views + Controllers (MonoBehaviour) | References Core via GUID |
| `Cloud`      | PlayFab auth + save                   | Interface + concrete; `IPlayFabDataWrapper` decouples PlayFab SDK for tests |
| `Input`      | Platform input adapters               | Flat-screen only — VR hand tracking lives in XR |
| `XR`         | Meta Quest 3 (optional)               | `excludePlatforms: ["WebGL"]` |
| `Tests`      | EditMode unit tests                   | `includePlatforms: ["Editor"]`, requires `UNITY_INCLUDE_TESTS` |
| `PlayModeTests` | PlayMode tests                     | Same refs as Tests |
| `Editor`     | Editor tools                          | |

Adding a new engine, model, or util → `Core/`. Adding a screen/UI/controller → `Presentation/`. **Do not** put `MonoBehaviour` in `Core/`, and **do not** put engine rules in `Presentation/`.

### DI root

`Assets/Project/Scripts/Architecture/DI/GameLifetimeScope.cs` (lives inside the `Core` asmdef, not its own). Extends VContainer's `LifetimeScope`. Engines are `Transient` (fresh per scene), cloud services are `Singleton`, controllers use `RegisterEntryPoint<T>` to bind into Unity's lifecycle via `IStartable`/`IDisposable`. **No `GameManager.Instance`, no `FindObjectOfType`** — everything goes through the container.

### Testability pattern

Production code that talks to external SDKs (PlayFab, future LootLocker, XR) takes an `I*Wrapper` interface in its constructor so unit tests inject a mock. See `PlayFabDataService` ↔ `IPlayFabDataWrapper` ↔ `MockPlayFabDataWrapper` (in `Tests/EditMode/Cloud/`). Async APIs return `Task<T>` via `TaskCompletionSource` callbacks — preserve that pattern.

## Conventions

- **Namespace pattern**: `CardFramework.<Layer>.<Sub>` — `CardFramework.Core.Engines`, `CardFramework.Cloud.PlayFab`, `CardFramework.Presentation.Controllers`, `CardFramework.Architecture.DI`.
- **DI everywhere.** New service = new interface + `Register<I,T>(Lifetime.X)` in `GameLifetimeScope.Configure`.
- **Controllers are POCO `IStartable/IDisposable`, not MonoBehaviour.** They subscribe to view events in `Start` and unsubscribe in `Dispose` — see `BlackjackTableController` for the canonical pattern.
- **Test naming**: `<ClassName>Tests` in `Tests/EditMode/<Layer>/` (or `Tests/PlayMode/...` for integration). Mocks are suffixed `Mock` and live alongside the test that uses them.
- **Async**: `TaskCompletionSource<T>` bridging callback-based SDKs to `Task<T>`. Never `.Result` or `.Wait()` — it deadlocks Unity.
- **Branch**: `feature/<epic>-<desc>` (current: `feature/ui-presentation-architecture`). Commit format `TASK-X.X: <verb> <thing>` (per README; not currently enforced).
- **Coverage**: 85%+ target (README claims 87% achieved in EPIC-01). Generate with `-enableCodeCoverage` + the `com.unity.testtools.codecoverage` package; reports land in `CodeCoverage/`.
